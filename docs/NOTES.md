# Notes ‚Äî Format, Schema & Authoring

This document is the single source of truth for note content: what fields a note has, how to write one, how links work, and how assistant-driven note creation must behave.

---

## Note Schema

Notes are stored in the `notes` table in Neon PostgreSQL. The schema source of truth is `src/lib/server/db/schema.ts`.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | uuid | auto | Generated by Postgres (`gen_random_uuid()`) |
| `title` | text | yes | Human-readable name. Must be unique. Drives the slug. |
| `slug` | text | auto | URL-safe, derived from title via `slugify()`. Unique. |
| `body` | text | no | Markdown. Use `[[Note Title]]` for internal links. |
| `tags` | text[] | no | From the Tag Taxonomy below. |
| `aliases` | text[] | no | Common alternate names for the topic. |
| `category` | text | no | One of the Hub Categories below. |
| `status` | enum | no | `stub` \| `growing` \| `mature`. Default: `stub`. |
| `ai_generated` | boolean | no | `true` if created or materially revised by an AI provider. Default: `false`. |
| `ai_model` | text | no | Model name (for example `claude-opus-4-6` or `gpt-4o`). Null if not AI. |
| `ai_prompt` | text | no | The originating assistant request or topic prompt. Null if not AI. |
| `created_at` | timestamp | auto | Set on insert. |
| `updated_at` | timestamp | auto | Must be set manually on update: `updatedAt: new Date()`. |

Graph edges are stored separately in `note_links` (source_note_id ‚Üí target_note_id) and are synced from `[[wikilinks]]` in note bodies on save.

---

## Status Values

| Status | Meaning | When to set |
|--------|---------|-------------|
| `stub` | Placeholder ‚Äî title and maybe a sentence | New or lightly researched notes |
| `growing` | Partially filled ‚Äî some sections complete | Actively being expanded |
| `mature` | Comprehensive ‚Äî all sections substantive | Reviewed and complete |

## Tag Taxonomy

Use one or more tags per note. Tags represent the broadest classification.

| Tag | Use for |
|-----|---------|
| `language` | Programming languages |
| `framework` | Frameworks and libraries |
| `service` | APIs, SaaS, cloud services |
| `tool` | CLIs, editors, dev tools, utilities |
| `concept` | Abstract ideas (RAG, microservices, CI/CD) |
| `model` | Specific AI/LLM models |
| `platform` | Broad platforms (AWS, GCP, Vercel, GitHub) |
| `protocol` | Standards and protocols (REST, gRPC, OAuth) |

## Wikilink Syntax

Use `[[Note Title]]` anywhere in the body to link to another note. The title must match exactly (case-sensitive).

**What happens on save:**
1. `extractWikilinks(body)` parses all `[[...]]` patterns from the body.
2. Each title is looked up in the `notes` table.
3. Resolved matches are written to `note_links` (source ‚Üí target).
4. Old links from that note are deleted first, then re-inserted (full re-sync).

**What happens on render:**
1. `resolveWikilinks(body, slugMap)` replaces `[[Title]]` with `<a href="/notes/slug" class="wikilink">Title</a>`.
2. Unresolved titles (note doesn‚Äôt exist) become `<span class="wikilink-broken">Title</span>` ‚Äî styled red with strikethrough.
3. The resolved body is passed to `marked()` for Markdown rendering.

**Assistant-first linking rule:**
- When the assistant creates a new note, the note body must already contain the `[[wikilinks]]` needed for immediate graph edges on save.
- If the assistant determines that one or more existing notes should also link to the new note, it should propose or perform those body updates in the same confirmed save flow so the graph reflects the connection immediately in both directions where appropriate.

**Rename propagation:**
- Renaming a note title automatically propagates the title change to `[[wikilinks]]` in all other notes‚Äô bodies before re-syncing `note_links`.

## Note Template

All notes should follow this structure:

```markdown
---
tags: []
aliases: []
category:
status: stub
---

# Topic Name

> [!summary]
> One sentence description.

## Overview
2-4 sentences. What it is, why it matters right now.

## Industry Usage
How it's used in modern development. What problems it solves. Current adoption and relevance.

## In Depth
Comprehensive explanation ‚Äî internals, tradeoffs, nuances, how it compares to alternatives.

## Connections
### Broader Context
[[Hub Note]] ‚Äî [[Another Category]]

### Related Topics
[[Related Thing]] ‚Äî [[Adjacent Concept]]

## Resources
- [Official Docs]()
```

Assistant-generated notes should still conform to this structure, even when the draft is reviewed and saved entirely from chat.

## Assistant Authoring Rules

1. **Assistant-first creation** ‚Äî new notes are expected to be proposed from chat rather than a dedicated `/notes/new` page.
2. **Live research first** ‚Äî assistant-created notes should be based on live web research plus the existing graph context.
3. **Populate required fields** ‚Äî the assistant must fill `title`, `body`, `tags`, `aliases`, `category`, and `status` before the note is confirmed for save.
4. **Link inline** ‚Äî use `[[wikilinks]]` inside prose, not only in the Connections section.
5. **Create immediate graph visibility** ‚Äî ensure the created note‚Äôs body contains the links needed for graph edges to appear as soon as the note is saved.
6. **Propagate relevant backlinks** ‚Äî if the new note should be referenced by existing notes, the assistant should also update those note bodies so reciprocal graph connections are visible immediately after save.
7. **Set status honestly** ‚Äî `stub` for minimal drafts, `growing` for partially complete research notes, `mature` only for strong coverage.
8. **Keep citations out of schema** ‚Äî live-web sources may be shown in chat review, but they are not persisted as dedicated source metadata in this phase.

---

## Human Authoring Rules

1. **Check before creating** ‚Äî search for the topic name and common aliases. Expand an existing note rather than duplicate it.
2. **Create hub notes** ‚Äî if a note‚Äôs category has no hub note, create it.
3. **Populate all frontmatter** ‚Äî always set `tags`, `aliases`, and `category`, even on stubs.
4. **Expand before adding sections** ‚Äî fill out the `In Depth` section before adding extra structure.
5. **Update aliases** ‚Äî add alternate names to frontmatter if the topic has common synonyms.
